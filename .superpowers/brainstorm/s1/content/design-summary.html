完整设计总结 - ecommerce-analyzer

逐节确认，每节通过后进入下一节

一、技术栈与模型策略

层V1 选型V2 补充
AI 主模型DeepSeek（文本推理/nl2sql）不变
视觉模型不需要通义千问VL / GPT-4o（chart_reader）
Web 框架Streamlit
数据处理Pandas + SQLite
可视化Plotly（交互优先）
预测—Prophet

二、多数据源抽象层

BaseDataSource 抽象类 → RetailRocketSource / CSVGenericSource 实现，统一输出标准 schema（user_id, item_id, event_type, timestamp）。后续新数据源只需实现子类。

三、看板布局

看板页面结构
📊 概览 👤 用户 🕳 漏斗 ⚠ 异常
图表区域（全屏沉浸）
AI 点击展开 AI 对话分析...

展开后为底部抽屉，分析结果支持导出 Markdown/PDF

对话持久化

SQLite 存储结构：

sessions（会话ID, 时间, 标题）
messages（消息ID, session_id, 角色, 文本）
artifacts（产物ID, msg_id, 类型, 路径）
← 图表/表格/报告都关联到对应消息

回溯时完整还原：对话 + 当时生成的图表 + 数据表

四、AI Agent 链路

用户："上周转化率为什么跌了？"
→ Agent 规划 → query_tools.get_kpi → drill_down(渠道) → nl2sql(自定义下钻)
→ insight_generator 汇总 → 返回报告（原因 + 数据 + 图表）
用户追问："移动端具体哪个环节掉的？"
→ Agent 继承上下文，继续下钻 → 漏斗按步骤拆解

五、开发顺序

板块范围版本
P0项目骨架：目录/配置/依赖/数据源抽象基类V1
P1数据管道：加载/清洗/校验/特征工程V1
P2EDA：画像/漏斗/留存/RFM/异常检测(统计)V1
P3看板 + 聊天底座 + 报告导出V1

P6新数据源接入验证 + 文档收尾V3
